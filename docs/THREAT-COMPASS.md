# Threat Compass: анализ уязвимостей веб-приложения

## 1. Архитектура

```
┌─────────────┐     HTTPS      ┌─────────────────┐     REST      ┌─────────────┐
│   Browser   │ ──────────────►│  React Frontend │ ─────────────►│   REST API  │
│   (User)    │                │  (SPA)          │               │  (Express)  │
└─────────────┘                └────────┬────────┘               └──────┬──────┘
                                        │                               │
                                        │ localStorage                  │
                                        │ (userId / будущий JWT)        │
                                        │                               ▼
                                        │                        ┌─────────────┐
                                        │                        │  Database   │
                                        │                        │ (YDB/PG)    │
                                        │                        └──────┬──────┘
                                        │                               │
                                        │                        ┌──────▼──────┐
                                        └────────────────────────│  S3 / files │
                                         (ссылки на файлы)       │ (если есть) │
                                                                 └─────────────┘
```

**Потоки данных:**
- Пользователь → Frontend: логин/пароль, формы (события, заявки, профиль).
- Frontend → API: все запросы без заголовка `Authorization` (в текущей реализации); в целевом сценарии — JWT в `Authorization: Bearer <token>`.
- API → DB: чтение/запись Users, Events, Applications, Notifications, EventRequests, Categories.
- API → S3 (если добавлено): загрузка/скачивание файлов по `imageUrl` или аналогу.

## 2. Активы (Assets)

| Актив | Расположение | Конфиденциальность |
|-------|--------------|--------------------|
| **Учётные данные** | БД (Users.password) | Критично — сейчас хранятся в открытом виде |
| **Идентификатор сессии / JWT** | Браузер (localStorage: `volunteer_app_user_id` или токен) | Критично — подмена даёт доступ от имени пользователя |
| **Персональные данные** | БД: Users (email, name, phone, avatar) | Высокая |
| **Заявки (Applications)** | БД | Высокая — привязка к userId/eventId |
| **Мероприятия (Events)** | БД | Средняя — часть публичная, часть черновики |
| **Запросы на создание событий (EventRequests)** | БД | Высокая |
| **Загруженные файлы** | S3 или внешние URL (поле `imageUrl`) | Средняя — при добавлении S3 возможна утечка через публичный bucket |

---

## 3. Границы доверия (Trust Boundaries)

1. **Интернет ↔ Frontend**  
   Ненадёжный ввод: всё, что вводит пользователь или что приходит в ответах API и отображается в UI (XSS, открытые редиректы).

2. **Frontend ↔ API**  
   Сейчас API не проверяет, кто вызывает запрос — граница не защищена. Целевая: проверка JWT/session на каждом запросе.

3. **API ↔ Database**  
   Внутренняя граница; доверие к API. Риск: SQL/NoSQL-инъекции, избыточный доступ к таблицам.

4. **API ↔ S3 (если есть)**  
   Облачное хранилище: риск публичного bucket, отсутствия проверки прав при выдаче presigned URL или при прямой отдаче файлов.

## 4. Типовые угрозы (по Threat Compass)

### 4.1 Аутентификация и сессии

| Угроза | Описание | Текущее состояние в коде |
|--------|----------|---------------------------|
| **Нет проверки авторизации на API** | Любой клиент может вызывать любые эндпоинты без токена/сессии | **Подтверждено**: в `server.js` нет middleware для `Authorization` или проверки пользователя |
| **Слишком длинный TTL токенов** | Украденный JWT действует долго | При введении JWT нужно задать короткий TTL и refresh |
| **Хранение паролей в открытом виде** | Утечка БД раскрывает все пароли | **Подтверждено**: в `server.js` сравнение `user.password !== password`, пароль в БД в plain text |
| **Идентификатор сессии только на клиенте** | Подмена `userId` в localStorage или запросе даёт доступ от имени другого пользователя | **Подтверждено**: API не проверяет, что запрос идёт от владельца ресурса |

### 4.2 IDOR (Insecure Direct Object Reference)

| Угроза | Эндпоинт/ресурс | Проверка |
|--------|------------------|----------|
| **Чтение/изменение чужого пользователя** | `GET/PATCH /users/:id` | Нет проверки: любой может запросить или изменить любого пользователя по `id` |
| **Чтение/изменение/удаление чужой заявки** | `GET/PATCH/DELETE /applications/:id` | Нет: можно менять статус или удалять чужую заявку |
| **Изменение чужого мероприятия** | `PATCH /events/:id` | Нет: любой может менять событие, в т.ч. organizerId |
| **Чтение/изменение заявок на создание событий** | `GET/PATCH /eventRequests/:id` | Нет привязки к организатору на уровне API |
| **Чтение чужих уведомлений** | `GET /notifications?userId=...` | Фильтр только по query; без авторизации можно подставить любой `userId` |

Пример ручной проверки IDOR заявок:
- Создать заявку от пользователя A, получить `id`.
- От имени пользователя B (или без авторизации, если API позволит) вызвать `PATCH /applications/{id}` с `status: 'approved'` или `DELETE /applications/{id}`.

### 4.3 XSS (Cross-Site Scripting)

| Вектор | Риск | Текущее состояние |
|--------|------|--------------------|
| **Поля событий/профиля** | title, description, location, schedule и т.д. отображаются в React как текст | **Снижен**: React по умолчанию экранирует; явного `dangerouslySetInnerHTML` для пользовательского контента нет |
| **imageUrl** | Использование в `<img src={event.imageUrl}>` | Риск: `javascript:...` или `data:` с вредоносным контентом; в части браузеров возможно выполнение. Нужна валидация схемы (только `https:`) |
| **SVG в Icon** | `dangerouslySetInnerHTML` для статических иконок | Иконки из бандла, не из API — риск низкий при текущей схеме |

Ручная проверка: в полях создания события (title, description) ввести `<script>alert(1)</script>` или `<img src=x onerror=alert(1)>` и убедиться, что вывод — экранированный текст, а не выполнение скрипта.

### 4.4 Конфигурация хранилища (S3)

При появлении S3 для загрузки изображений/файлов:

| Угроза | Описание |
|--------|----------|
| **Public bucket** | Файлы доступны по прямой ссылке без проверки прав — чтение чужих загрузок |
| **Отсутствие проверки при выдаче URL** | Presigned URL или ссылка выдаются без проверки, что пользователь имеет право на этот объект |
| **Загрузка без проверки типа/размера** | Риск загрузки исполняемых файлов, переполнения хранилища |

Ручная проверка: загрузить файл, получить URL, открыть в режиме инкогнито без авторизации — доступ не должен быть возможен для приватных объектов.

### 4.5 Rate limiting и доступность

| Угроза | Описание | Текущее состояние |
|--------|----------|-------------------|
| **Отсутствие rate limiting на API** | DoS, перебор паролей по списку email (если эндпоинт не защищён) | На `/auth` есть ограничение: 5 неудачных попыток в час → блок 15 мин по email. Остальные эндпоинты без лимитов |
| **Массовые запросы на чтение** | GET /users, GET /applications и т.д. без лимита | Возможен перебор ID и скачивание данных |

### 4.6 CORS и сетевая конфигурация

| Угроза | Текущее состояние |
|--------|--------------------|
| **Слишком открытый CORS** | `corsOrigin = process.env.CORS_ORIGIN || '*'` — в продакшене нужно явно задать origin фронтенда |

## 5. Чек-лист ручной проверки

- [ ] **IDOR заявок**: под другим пользователем (или без токена, если API начнёт принимать анонимные запросы) вызвать `GET /applications/{чужой_id}` и `PATCH /applications/{чужой_id}` — доступ должен быть запрещён.
- [ ] **IDOR пользователей**: `GET /users/{чужой_id}` и `PATCH /users/{чужой_id}` — только свой профиль или только с правами админа.
- [ ] **Файл без авторизации**: при наличии S3 — ссылка на приватный объект не должна открываться без валидной авторизации/signature.
- [ ] **XSS**: в полях ввода (имя, описание события, сообщение в заявке) вставить `<script>alert(1)</script>` — нигде не должно выполняться.
- [ ] **Пароли**: убедиться, что в БД хранятся только хеши (bcrypt/argon2), не plain text.
- [ ] **Токены**: при переходе на JWT — проверить TTL, наличие refresh, передачу только по HTTPS и в заголовке (не в URL).

## 6. Рекомендуемые меры (кратко)

1. **API**: ввести middleware аутентификации (JWT или сессия), проверять на каждом запросе к защищённым ресурсам.
2. **Авторизация**: на эндпоинтах вида `/users/:id`, `/applications/:id`, `/events/:id`, `/eventRequests/:id` проверять, что `id` принадлежит текущему пользователю или что у пользователя роль admin/organizer где уместно.
3. **Пароли**: хранить только хеши (bcrypt/argon2); при логине сравнивать с `bcrypt.compare`.
4. **imageUrl**: валидировать на бэкенде (схема `https:`, при необходимости — allowlist доменов).
5. **Rate limiting**: общий лимит на API (например, по IP или по токену) и отдельно — на логин (уже есть по email).
6. **CORS**: в продакшене задать `CORS_ORIGIN` явно (один или несколько origin фронтенда).
7. **S3**: приватный bucket; доступ только через presigned URL с коротким TTL; проверка прав перед выдачей URL.

После внедрения изменений — повторить пункты чек-листа и при необходимости обновить этот документ.
